깃 커밋 컨벤션
목표
현재 워킹트리의 실제 변경사항을 직접 확인한 뒤, 적절한 커밋 단위로 나눠 실행 가능한 명령어만 출력한다.

확인 절차
커밋 전 반드시 아래 순서로 직접 확인한다.

git status --short — 변경된 파일 목록 파악
git diff --name-only — 수정 범위 확인
git diff -- <파일경로> — 파일별 실제 변경 내용 확인
확인한 diff 기준으로만 커밋 단위를 나눈다. 파일 목록을 따로 받지 않아도 직접 확인해서 판단한다.

커밋 분리 기준
파일이 달라도 하나의 기능·사용자 흐름이면 한 커밋으로 묶는다.
같은 파일 안이어도 목적이 다르면 커밋을 분리한다.
포맷팅·개행 정리 같은 부수 수정은 관련 기능 커밋에 포함한다.
diff로 직접 확인하지 않은 변경은 절대 포함하지 않는다.
커밋 메시지 형식

type(scope): 제목
type — feat / fix / refactor / chore 중 실제 변경 성격에 맞게 선택

scope — 슬래시 없이 페이지명·기능명만 짧게

예: party_form, consultation_form, status_badge

제목 — 한글, 구체적으로, 결과가 바로 보이게

내부 구현 용어 단독 사용 금지 (프리필, 리팩토링, 공통화 등)
누가 봐도 무엇이 달라졌는지 알 수 있어야 함
예: 전화번호로 당사자 신규 등록 화면 진입 지원

본문 — 항상 작성, 두 번째 -m으로 전달

각 줄은 - 설명 형식
실제 줄바꿈 사용 (\n 문자열·쉼표 나열·한 줄 몰아쓰기 금지)
출력 규칙
명령어만 출력한다. 설명·해설·선택지·부연 설명 없음
커밋마다 git add ... 바로 다음 줄에 git commit ...
커밋이 여러 개면 순서대로 반복
금지사항
예시용·가정용·추측성 커밋 명령어
이전 응답에서 사용한 커밋 제목 재사용
실제 diff와 맞지 않는 제목·본문
본문 없는 커밋
모호하거나 너무 짧은 제목
출력 예시

git add frontend/lib/pages/party/party_form_screen.dart
git commit -m "feat(party_form): 전화번호로 당사자 신규 등록 화면 진입 지원" -m "- 전화번호가 포함된 진입 경로를 추가
- 신규 등록 화면에서 대표연락처를 자동 입력
- 잘못된 값은 당사자 목록으로 이동하도록 처리"
